- 이유: 이전 observation은 lifecycle/terminal 입력 리포트가 오래됐는데도 fresh report처럼 섞어 판단했고, 전체 run 실패가 1건이라도 있으면 source health가 healthy여도 `hold_health`로 떨어졌다.
- 변경: 입력 report별 age/freshness를 JSON/Markdown에 표시하고, stale 입력이 있으면 `hold_stale_inputs`로 분리한다. health hold 조건은 source health, tick 실패, 전체 실패율 기준으로 좁혔다.
- 재검증: tick/db/pack/lifecycle/terminal report를 모두 fresh로 재생성한 뒤 observation을 다시 실행했다.
- 결과: fresh 입력 기준으로도 `hold_health`가 맞다. 최신 24h window는 400 runs 중 21 failed(5.3%), source health `degraded`, pack open 38/38이다.
- 원인: 실패 21건은 모두 Supabase `23514` check constraint 위반이고, `mvp_raw_listings.listing_type`에 DB 허용값이 아닌 내부 triage label `title_sku_match`가 저장되면서 발생했다. 반복 대상 pid는 `407645878`로 확인했다.
- 변경: `src/lib/tick-pipeline.ts`에 raw 저장용 listing_type 정규화 함수를 추가했다. 내부 triage label은 관측 JSON에는 남기되, DB `listing_type`에는 허용된 값만 저장한다. `title_sku_match`는 저장 시 `unknown`으로 내려간다.
- 검증: `npx eslint src/lib/tick-pipeline.ts --max-warnings=0` 통과, `npx tsx --test tests/core-rules.test.ts tests/ai-l2-policy.test.ts` 통과(90 tests).
- 보류: DB check constraint 변경, raw row 수동 보정, DDL/RPC batching은 하지 않았다.
- 다음: 다음 자연 tick/deep_crawl 이후 `23514` 실패가 재발하는지 `report:db-hotpaths`로 확인한다. 재발하지 않으면 raw touch RPC batching보다 먼저 source health recovery window를 본다.
