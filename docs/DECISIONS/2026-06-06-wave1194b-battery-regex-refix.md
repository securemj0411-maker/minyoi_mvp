## 발견 (audit agent 재현 테스트로 확정)

Wave 1194의 배터리 fix가 **실제로 작동 안 했음 + 신규 FP 유발**:

1. **죽은 코드**: `normalize()`(option-parser.ts:477 `.replace(/[^0-9a-z가-힣./\s]+/g, " ")`)가
   정규식 실행 **전에 `%`를 제거**. 그런데 Wave 1194 fallback #4가 리터럴 `%`를 요구 → 절대 발화 불가.
   → owner 케이스 "베터리 용량 64%"가 normalize 후 "베터리 용량 64"가 되어 여전히 **null**.
   (Wave 1194의 12케이스 테스트가 normalize를 안 거친 입력이라 통과처럼 보였던 것 — 내 검증 실수.)

2. **신규 FP**: 패턴 #1이 배터리 키워드 optional + `%?` optional + 숫자 `[1-9][0-9]`로 넓어져서
   "노트북 성능 60", "발열 성능 30", "수명 95점", "그래픽 성능 99" 등 **비-배터리 숫자를
   batteryHealth로 오인** → batteryHealth<85면 `low_batt` → 정상 전자기기가 열화 시세군으로 오분류 + 페널티.

## fix (재설계)

```
/(?:배터리|베터리|밧데리|빳데리|바테리)[^.!?\n]{0,20}?(100|[1-9][0-9])(?!\d)(?!\s*(?:충전|남|정도|기가|gb|메가|인치|만|원|개|회))/
/배효\s*[:：]?\s*(100|[1-9][0-9])(?!\d)/
/신품\s*대비\s*(100|[1-9][0-9])(?!\d)/
```

- **% 없이** 매칭 (normalize가 % 지우므로 % 의존 불가).
- **배터리류 키워드 필수** → "성능 60"/"수명 95점" 등 FP 차단 (키워드 없으면 안 잡음).
- 단위(기가/gb/메가/인치/만/원/개/회) + 충전잔량(충전/남/정도) trap 배제.

### 테스트 (실제 normalize 거쳐) 15/15 통과
- 잡음: "베터리 용량 ... 64%"=64, "배터리 성능은 88%"=88, "배터리효율 78%"=78, "배효 92"=92, "배터리 100%"=100, "신품대비 80%"=80, "밧데리 성능 65프로"=65
- 차단(null): "노트북 성능 60", "발열 성능 30", "수명 95점", "그래픽 성능 99", "가성비 성능 85", "배터리 50% 충전", "저장 용량 256기가", "배터리 256기가"
